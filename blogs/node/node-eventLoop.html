    # node EventLoop阶段

    1. timers : 执行setTimeOut和setInteval 的回调
    2. pending callback: 执行延迟到下一个循环迭代的I/O回调
    3. idle,prepare: 仅系统内部使用
    4. poll: 检索新的I/O事件； 执行与I/O相关的回调事件， 事实上除了其他几个阶段处理的事情，其他几乎所有的异步都在这个阶段处理。
    5. check : setImmediate在这里执行
    6. close callbacks: 一些关闭的回调函数，如：socket.on('close', ...)

    # node比浏览器多了两个api

    process.nextTick： 严格上来讲不属于事件循环中，可以看做微任务， 始终比其他的微任务先执行，在 每个阶段前执行

    setImmediate: 宏任务

    • 在主线中与setTimeout的执行顺序是不确定的，取决的当前计算机的进程性能。
    • 在I/O回调中和times （setTimeout）中的 setImmediate始终是在setTimeout之前运行

    从名称上和表现上，process.nextTick 与 setImmediate 应该换一个名字，由于历史原因，所有已发布的包会受影响，所以只能如此。。。

    setInterval是setTimerout的一个语法糖，在每一次时间到达的时候加入到timers队列中。

    # 为什么还用 process.nextTick

    1. 某些场景下需要让回调在事件循环下一个阶段前执行

    比如： 监听listening，如果使用setImmdiate的话那么说明poll阶段执行了，可能已经接收到连接，而listening还未执行，导致漏掉一些事件！！
